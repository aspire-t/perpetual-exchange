## 🔴 高危问题 (High Severity)

### 问题 1: 私钥硬编码在 .env 文件中

**位置**: `backend/.env`, `frontend/.env.local`

**问题描述**:
```env
# backend/.env
HYPERLIQUID_PRIVATE_KEY=0x25c86c1b938d513e89579f42cb1c527f3c179f6b1b9834a03ab880858bc5f10a
MNEMONIC="test test test test test test test test test test test junk"

# frontend/.env.local
NEXT_PUBLIC_TESTNET_PRIVATE_KEY=0x25c86c1b938d513e89579f42cb1c527f3c179f6b1b9834a03ab880858bc5f10a
```

**影响分析**:
- 私钥明文存储在版本控制系统中
- 测试网私钥被团队共享，任何成员都可以控制该地址资金
- 如果部署到生产环境，存在资金被盗风险
- 违反了安全最佳实践：私钥应该使用密钥管理服务（如 AWS Secrets Manager, HashiCorp Vault）

**修复建议**:
1. 将私钥从 .env 文件中移除
2. 使用环境变量注入或密钥管理服务
3. 将 .env 添加到 .gitignore
4. 提供 .env.example 模板文件

### 问题 2: 前端/后端 Vault 合约地址不一致

**位置**:
- `frontend/.env.local`: `NEXT_PUBLIC_VAULT_ADDRESS=0xDd91aA72d6c88D270036036C445727b3D8967deE`
- `backend/src/indexer/event-listener.service.ts`: 默认地址 `0xe7f1725E7734CE288F8367e1Bb143E90bb3F0512`

**问题描述**:
前端和后端配置的 Vault 合约地址不一致，导致：
- 前端用户向一个合约充值
- 后端监听另一个合约的事件
- 充值事件无法被正确捕获，用户余额无法更新

**影响分析**:
- 用户充值后无法到账
- 系统资金流转中断
- 严重的功能缺陷

**修复建议**:
1. 统一前后端 Vault 合约地址配置
2. 使用部署脚本输出合约地址到共享配置文件
3. 在 CI/CD 流程中自动同步地址配置

### 问题 3: 对冲失败时订单执行风险暴露

**位置**: `backend/src/order/order.service.ts` 第 447-455 行

**问题描述**:
```typescript
if (!hedgeResult.success || !hedgeResult.data) {
  // Refund margin and fee if hedge fails
  await this.balanceService.releaseMargin(user.id, marginRequiredUSDC);
  await this.balanceService.releaseMargin(user.id, BigInt(0), tradingFeeUSDC);
  return { success: false, error: `Hedge execution failed: ${hedgeResult.error}` };
}
```

当前代码在对冲失败时退还保证金和手续费，但：
- 订单状态被标记为 PENDING，未显式标记为 REJECTED
- 如果对冲成功但创建仓位失败，存在风险敞口

**更严重的是**: 代码注释提到"假设我们需要退还手续费"，但实际上 `releaseMargin(user.id, BigInt(0), tradingFeeUSDC)` 是将手续费作为 PnL 加回，这不是正确的退款逻辑。

**影响分析**:
- 极端情况下可能导致资金损失
- 订单状态不一致
- 财务记录不准确

**修复建议**:
1. 明确订单状态为 REJECTED
2. 实现对冲成功但仓位创建失败的回滚逻辑
3. 添加事务一致性保证

### 问题 8: weiToTokenSize 假设所有代币都是 18 位小数

**位置**: `backend/src/hedging/hedging.service.ts` 第 481-484 行

**问题描述**:
```typescript
private weiToTokenSize(weiSize: bigint): string {
  const decimals = BigInt('1000000000000000000'); // 18 decimals
  return (weiSize / decimals).toString();
}
```

**影响分析**:
- BTC 通常是 8 位小数
- 某些稳定币是 6 位小数（如 USDC）
- 导致对冲数量计算错误

**修复建议**:
1. 为每个 symbol 配置正确的小数位数
2. 从 Chainlink 或合约动态获取小数位
3. 实现 symbol 到 decimals 的映射配置

## 📝 总结

现有代码已实现永续合约交易所的核心功能闭环（充值→交易→对冲→平仓），但在以下方面需要改进：

1. **安全性**: 私钥管理、签名验证方式需要加强
2. **一致性**: 前后端配置需要对齐
3. **健壮性**: 异常场景处理需要完善
4. **可维护性**: 测试覆盖率需要提升

建议优先修复高危问题，确保系统安全稳定运行。
