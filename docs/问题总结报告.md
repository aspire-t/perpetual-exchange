### 问题 1: 私钥硬编码在 .env 文件中

**位置**: `backend/.env`, `frontend/.env.local`

**问题描述**:
```env
# backend/.env
HYPERLIQUID_PRIVATE_KEY=0x25c86c1b938d513e89579f42cb1c527f3c179f6b1b9834a03ab880858bc5f10a
MNEMONIC="test test test test test test test test test test test junk"

# frontend/.env.local

```

**影响分析**:
- 私钥明文存储在版本控制系统中
- 测试网私钥被团队共享，任何成员都可以控制该地址资金
- 如果部署到生产环境，存在资金被盗风险
- 违反了安全最佳实践：私钥应该使用密钥管理服务（如 AWS Secrets Manager, HashiCorp Vault）

**修复建议**:
1. 将私钥从 .env 文件中移除
2. 使用环境变量注入或密钥管理服务
3. 将 .env 添加到 .gitignore
4. 提供 .env.example 模板文件

### 问题 3: 对冲失败时订单执行风险暴露

**位置**: `backend/src/order/order.service.ts` 第 447-455 行

**问题描述**:
```typescript
if (!hedgeResult.success || !hedgeResult.data) {
  // Refund margin and fee if hedge fails
  await this.balanceService.releaseMargin(user.id, marginRequiredUSDC);
  await this.balanceService.releaseMargin(user.id, BigInt(0), tradingFeeUSDC);
  return { success: false, error: `Hedge execution failed: ${hedgeResult.error}` };
}
```

当前代码在对冲失败时退还保证金和手续费，但：
- 订单状态被标记为 PENDING，未显式标记为 REJECTED
- 如果对冲成功但创建仓位失败，存在风险敞口

**更严重的是**: 代码注释提到"假设我们需要退还手续费"，但实际上 `releaseMargin(user.id, BigInt(0), tradingFeeUSDC)` 是将手续费作为 PnL 加回，这不是正确的退款逻辑。

**影响分析**:
- 极端情况下可能导致资金损失
- 订单状态不一致
- 财务记录不准确

**修复建议**:
1. 明确订单状态为 REJECTED
2. 实现对冲成功但仓位创建失败的回滚逻辑
3. 添加事务一致性保证

### 问题 5: 核心接口未鉴权，完全信任请求体中的地址（可替他人下单/提现）

**位置**:
- 前端: 缺少签名登录逻辑，仅传递连接的钱包地址。
- 后端: `backend/src/order/order.controller.ts` 和 `backend/src/withdrawal/withdrawal.controller.ts`

**问题描述**:
核心业务接口（如下单 `/order`、提现 `/withdraw`）没有使用 JWT 或任何 Guard 保护，而是直接从 `@Body() body: CreateOrderDto` 中提取 `address` 字段执行业务。
前端也没有调用 `/auth/login` 进行签名验证，导致身份验证形同虚设。

**影响分析**:
- 任何人都可以通过 API 伪造任意地址，替他人下单、平仓甚至发起提现。
- 极其严重的安全漏洞，资金可被轻易耗尽。

**修复建议**:
1. 后端核心接口必须添加 `@UseGuards(JwtAuthGuard)`。
2. 从 `req.user.address` 中获取当前用户地址，严禁信任客户端传入的 `address` 字段。
3. 前端完整实现 EIP-712 签名登录，并在 HTTP 请求头中携带 JWT Token。

### 问题 6: 核心业务缺乏事务保证，存在严重并发与一致性漏洞

**位置**: `backend/src/order/order.service.ts` 的 `executeOrder` 方法

**问题描述**:
在执行订单时，代码顺序调用了 `lockMargin`、`deductFee`、`executeHedgeOrder`、`increasePosition`/`openPosition` 和 `orderRepository.save`。
这些操作没有被包裹在同一个数据库事务（Transaction）中。

**影响分析**:
- 极其脆弱的并发处理能力。如果服务器在对冲成功后、保存订单或创建仓位前崩溃，会导致：用户保证金被扣除，Hyperliquid 上的对冲仓位已建立，但本地数据库没有仓位记录（链下状态与第三方对冲状态严重不一致）。
- 资金可能处于被永远锁定（无法释放）的状态。

**修复建议**:
1. 使用 TypeORM 的 `DataSource` 和 `QueryRunner` 将整个开仓/加仓逻辑包裹在单一事务中。
2. 确保对冲请求与本地数据库状态更新实现最终一致性（或使用两阶段提交思想/补偿机制处理外部 API 调用）。

## 🔴 Critical 问题 (严重等级最高)

### 问题 19: 对冲成功但创建仓位失败无回滚

**位置**: `backend/src/order/order.service.ts` 第 482-485 行、499-501 行

**问题描述**:
```typescript
if (!increaseResult.success) {
  // This is a rare error since we already hedged.
  this.logger.error(`Failed to increase position after hedge: ${increaseResult.error}`);
  return { success: false, error: increaseResult.error };
}
// ... 同样的问题存在于 openPosition 失败处理
if (!openResult.success) {
  this.logger.error(`Failed to open position after hedge: ${openResult.error}`);
  return { success: false, error: openResult.error };
}
```

当前代码在对冲成功后，如果创建/增加仓位失败，直接返回错误但**没有回滚已执行的对冲**。这导致:
- Hyperliquid 上的对冲仓位已建立
- 本地数据库没有对应的仓位记录
- 系统存在未跟踪的风险敞口

**影响分析**:
- 极端情况下可能导致资金损失
- 链下状态与第三方对冲状态严重不一致
- 无法通过内部系统管理或平仓该对冲仓位

**修复建议**:
1. 在 createHedgeRecord 失败时，调用 Hyperliquid API 平仓已建立的对冲
2. 或实现补偿机制，将孤立的对冲仓位标记为"待处理"并通知人工干预
3. 优先方案：使用事务包裹整个流程，外部 API 失败时回滚

### 问题 20: 订单执行流程无事务包裹

**位置**: `backend/src/order/order.service.ts` 第 350-543 行 (executeOrder 方法)

**问题描述**:
```typescript
async executeOrder(...) {

  // 2. 扣除手续费

  // 3. 执行对冲 (外部 API 调用)

}
```

上述多步操作**没有被包裹在同一个数据库事务中**。

**影响分析**:
- 极其脆弱的并发处理能力
- 如果服务器在对冲成功后、保存订单前崩溃，会导致:
  - 用户保证金被扣除
  - Hyperliquid 上的对冲仓位已建立
  - 本地数据库没有仓位记录和订单记录
- 资金可能处于被永远锁定的状态

**修复建议**:
1. 使用 TypeORM 的 `DataSource` 和 `QueryRunner` 将整个开仓/加仓逻辑包裹在单一事务中
2. 确保对冲请求与本地数据库状态更新实现最终一致性
3. 或使用两阶段提交思想/补偿机制处理外部 API 调用

**位置**: `backend/src/risk/risk-engine.service.ts` 第 523-524 行

**问题描述**:
```typescript
const shouldLiquidate =
  parseFloat(positionAtRisk.healthFactor) < this.LIQUIDATION_THRESHOLD;
```

`LIQUIDATION_THRESHOLD = 0.025` 但健康因子 (Health Factor) 的正常范围是:
- HF > 1: 健康
- HF < 1: 有清算风险
- HF 接近 0: 应被清算

使用 0.025 作为阈值意味着健康因子降到 0.025 才触发清算，这过于激进。

**影响分析**:
- 可能在健康因子还很高时不触发清算
- 导致系统风险积累

**修复建议**:
1. 使用合理的健康因子阈值 (如 0.8 或 0.9)
2. 或重新设计阈值常量命名，区分"预警线"和"清算线"
