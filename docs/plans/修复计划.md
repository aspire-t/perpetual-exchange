## Stage 1: 安全与核心一致性 (优先修复)

### 问题 1: 私钥硬编码在 .env 文件中

**根本原因**:
- 开发阶段为方便将私钥明文写入版本控制
- 缺少密钥管理意识和管理流程

**修复方案**:
1. 将 `.env` 添加到 `.gitignore`
2. 创建 `.env.example` 模板文件（含占位符）
3. 移除代码中所有硬编码私钥引用
4. 使用环境变量注入：`process.env.HYPERLIQUID_PRIVATE_KEY`
5. 生产环境建议接入 AWS Secrets Manager 或 HashiCorp Vault

---

### 问题 2: 前端/后端 Vault 合约地址不一致

**根本原因**:
- 前后端独立配置，缺少统一的配置管理
- 部署流程中未自动同步合约地址

**修复方案**:
1. 确认正确的 Vault 合约地址
2. 统一后端 `backend/.env` 中的 `VAULT_ADDRESS`
3. 统一前端 `frontend/.env.local` 中的 `NEXT_PUBLIC_VAULT_ADDRESS`
4. 建议：创建部署脚本输出合约地址到共享配置文件

---

### 问题 3: 对冲失败时订单执行风险暴露

**根本原因**:
- 退款逻辑不清晰，`releaseMargin` 参数使用错误
- 订单状态未正确更新为 `REJECTED`

**修复方案**:
1. 对冲失败时显式设置订单状态为 `REJECTED`
2. 修正退款逻辑：`releaseMargin(user.id, marginRequiredUSDC)` 仅退还保证金
3. 手续费应单独处理（退还或标记为已消耗）

---

### 问题 4: 项目 Vault 合约源码缺失

**根本原因**:
- 合约可能使用外部部署或独立仓库
- 未纳入当前版本控制

**修复方案**:
1. 确认 Vault 合约源码位置
2. 如使用外部合约：提供合约地址和区块浏览器验证链接
3. 如为自有合约：将源码纳入 `contracts/` 下与当前仓库结构一致的源码目录

---

### 问题 5: 核心接口未鉴权（可替他人下单/提现）

**根本原因**:
- 控制器直接从请求体提取 `address` 字段
- 缺少 JWT Guard 保护
- 前端未实现签名登录流程

**修复方案**:
1. 后端核心接口添加 `@UseGuards(JwtAuthGuard)`
2. 从 `req.user.address` 获取用户地址，禁止信任客户端传入的 `address`
3. 前端实现 EIP-712 签名登录，获取 JWT 后在请求头携带
4. 提现接口额外添加二次确认或冷却时间

**依赖**: 问题 8 (EIP-712 签名)

---

### 问题 6: 核心业务缺乏事务保证

**根本原因**:
- `executeOrder` 方法顺序执行多步操作，未使用 TypeORM 事务
- 外部 API 调用与数据库操作未实现原子性

**修复方案**:
1. 使用 `DataSource.createQueryRunner()` 创建事务
2. 将以下操作包裹在单一事务中：
   - 锁定保证金
   - 扣除手续费
   - 执行对冲
   - 创建/增加仓位
   - 创建对冲记录
   - 保存订单
3. 外部 API 失败时回滚所有数据库操作
4. 实现补偿机制：对冲成功但本地失败时，调用 Hyperliquid API 平仓

---

### 问题 19: 对冲成功但创建仓位失败无回滚

**根本原因**:
- 与问题 6 相同，缺少事务包裹
- 错误处理仅返回错误，未执行回滚

**修复方案**:
1. 此问题由问题 6 的事务修复一并解决
2. 在事务块中，任何步骤失败都会自动回滚
3. 额外实现：如对冲已成功，调用补偿机制平仓

**依赖**: 问题 6

---

### 问题 20: 订单执行流程无事务包裹

**根本原因**:
- 与问题 6 相同

**修复方案**:
- 与问题 6 合并修复

**依赖**: 问题 6

## Stage 2A: 精度与符号统一

### 问题 10: weiToTokenSize 假设所有代币都是 18 位小数

**根本原因**:
- 代码硬编码 `1000000000000000000` (18 decimals)
- 未考虑不同代币的精度差异

**修复方案**:
1. 创建 `TokenConfig` 配置类/文件：
   ```
   ETH: { decimals: 18, symbol: 'ETH' }
   BTC: { decimals: 8, symbol: 'BTC' }
   USDC: { decimals: 6, symbol: 'USDC' }
   ```
2. 修改 `weiToTokenSize(weiSize: bigint, symbol: string)` 接受 symbol 参数
3. 从配置中获取对应精度进行转换

---

### 问题 11: 前端 size 转换精度问题

**根本原因**:
- 前端使用 `parseEther` 固定 18 位精度
- 与后端精度处理不一致

**修复方案**:
1. 前端引入相同的 `TokenConfig`
2. 根据用户选择的交易对使用对应精度：
   ```typescript
   const sizeWei = parseUnits(size.toString(), tokenConfig.decimals)
   ```
3. 确保前后端精度转换逻辑一致

---

### 问题 22: 硬编码 ETH 符号

**根本原因**:
- 多处代码写死 `'ETH'` 而非从参数获取
- 缺少多币种支持设计

**修复方案**:
1. 识别所有硬编码位置：
   - `hedging.service.ts`: 149, 244, 255, 416
   - `position.service.ts`: 84
   - `risk-engine.service.ts`: 430, 512
2. 修改方法签名，添加 `symbol: string` 参数
3. 从 `Position` 或 `Order` 实体获取 `symbol` 字段
4. 实现 symbol 到 Hyperliquid 资产索引的映射

## Stage 2B: Indexer 与事件处理

### 问题 7: Indexer 服务无起始区块配置

**根本原因**:
- Filter 未配置 `fromBlock`
- 依赖节点默认行为

**修复方案**:
1. 添加环境变量 `START_BLOCK` 配置
2. 在 `IndexerService` 中读取并验证：
   ```typescript
   this.filter = {
     address: [this.contractAddress],
     topics: [],
     fromBlock: parseInt(process.env.START_BLOCK || '0'),
   }
   ```
3. 利用已有的 `ProcessedEvent` 表追踪处理进度

### 问题 9: Withdraw 事件处理未阻塞资金不足情况

**根本原因**:
- 仅记录警告，未跳过或失败处理
- 缺少异常流程控制

**修复方案**:
1. 资金不足时阻塞事件处理并记录失败：
   ```typescript
   if (lockedBalance < withdrawAmount) {
     this.logger.error(`Withdrawal blocked: insufficient locked balance, tx=${event.txHash}`)
     // 记录失败状态并终止本次事件写入
     return
   }
   ```
2. 发送告警通知到监控渠道
3. 标记人工复核状态（待处理）

## Stage 2C: 风险管理逻辑

### 问题 23: 健康因子比较逻辑错误

**根本原因**:
- `LIQUIDATION_THRESHOLD = 0.025` 过小
- 健康因子正常范围：>1 健康，<1 有风险

**修复方案**:
1. 修改清算阈值为合理值：
   ```typescript
   private readonly LIQUIDATION_THRESHOLD = 0.8 // 或 0.9
   ```
2. 或区分"预警线"和"清算线"：
   ```typescript
   private readonly WARNING_THRESHOLD = 1.0
   private readonly LIQUIDATION_THRESHOLD = 0.8
   ```

---

### 问题 24: 负 PnL 处理问题

**根本原因**:
- `releaseMargin` 中 `marginAmount + pnl` 未检查边界
- pnl 为负数且绝对值大于 margin 时余额为负

**修复方案**:
1. 添加边界检查：
   ```typescript
   const totalAmount = marginAmount + pnl
   if (totalAmount < 0) {
     // 处理不足部分：追缴或标记为负债
     this.logger.error(`PnL exceeds margin: ${pnl}`)
     totalAmount = 0 // 或实现追缴逻辑
   }
   userBalance.available += totalAmount
   ```

## Stage 2D: 其他逻辑修复

### 问题 8: 钱包签名验证未使用 EIP-712

**根本原因**:
- 使用简单的 `verifyMessage` 而非 `verifyTypedData`

**修复方案**:
1. 定义 EIP-712 类型化数据结构：
   ```typescript
   const domain = { name: 'PerpetualExchange', version: '1', chainId: 1 }
   const types = {
     Login: [{ name: 'address', type: 'address' }, { name: 'nonce', type: 'uint256' }]
   }
   ```
2. 后端使用 `ethers.verifyTypedData(domain, types, data, signature)`
3. 前端使用 `signer.signTypedData(domain, types, data)`
